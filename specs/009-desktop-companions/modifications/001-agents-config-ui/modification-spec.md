## Input
User description: "@agents/ config UI watch paths should have a file picker instead of just plain text input"

## Why Modify?
The desktop configuration companion already makes watch-directory changes approachable, but plain text-only path entry is still easy to mistype and requires users to know exact filesystem paths. A native file or directory picker for watched paths reduces configuration mistakes, improves discoverability for non-terminal users, and better matches the goal of safe local configuration editing without opening the full web surface.

## What's Changing?

### Added
- A user-visible file or directory picker affordance for each watch-path row in the configuration UI.
- A picker result flow that updates the selected watch path while preserving the existing validation and save behavior.
- Coverage expectations for choosing paths through the picker, canceling without changing the value, and retaining manually entered paths.

### Modified
- Configuration watch-path editing: **Was**: users edited watch paths only as plain text. **Now**: users can select paths through a picker and may still edit or correct paths manually when needed. **Why**: picker-based selection prevents path typos and makes watch-directory setup easier.
- Config editor validation: **Was**: validation only needed to handle values typed into text fields. **Now**: validation must handle values produced by the picker and preserve existing path edge-case handling for quotes, backslashes, duplicates, blank paths, and invalid TOML. **Why**: picker output must not bypass safety checks.
- Accessibility evidence: **Was**: config editor evidence covered labels, text, prompts, and diagnostics. **Now**: evidence must also cover the picker control label, keyboard access where supported by the GUI toolkit, and plain text fallback/error messaging. **Why**: the picker is a new user-facing control.

### Removed
- No user-facing capability is removed.
- Plain text path editing is not removed unless planning proves the GUI toolkit supplies an accessible equivalent for all required correction and unsupported-picker cases.

### Unchanged (Important to Document)
- The underlying configuration TOML shape remains unchanged: watch entries still store `path` strings and `patterns` arrays.
- Existing readable configuration structure and comments must still be preserved where possible.
- Required-field validation and plain text error diagnostics remain mandatory.
- Existing spine REST contracts, bearer-token capture behavior, local IPC status/reindex commands, quick capture behavior, tray behavior, queue storage, installers, and service/task integration remain unchanged unless implementation discovers a direct config companion dependency.
- Existing manually typed watch paths and existing config files must remain readable without migration.

## Updated User Scenarios
*Filled during /speckit.plan - Only scenarios that change, reference original spec for unchanged*

### Modified Scenario: Edit Configuration Safely
**Original**: A user edits spine connection details and watched directories through a local configuration companion using displayed editable settings, then saves and optionally requests reindex or restart.
**Modified**: **Given** an existing or new watch-directory row, **When** the user chooses the watch path picker and selects a filesystem path, **Then** the selected path is displayed in the config UI, validated through the same rules as typed paths, and saved to the existing TOML watch path field without changing unrelated comments or formatting where preservation is supported.
**Why Changed**: Picker-assisted selection reduces path-entry mistakes and makes watch-directory configuration easier for desktop users.

### New Scenario: Picker Cancel Preserves Existing Path
**Given** a watch-directory row with an existing path, **When** the user opens the picker and cancels or the picker cannot complete, **Then** the existing path remains unchanged and the user receives clear text feedback if action is required.
**Why Added**: Cancel/failure behavior must be safe and must not accidentally remove watch paths.

### Removed Scenario: None
**Was**: N/A
**Why Removed**: No original scenario is removed.
**Migration**: Existing manual path editing and existing TOML files continue to work.
